The Ansible control plane that doesn't need a control plane.

Pine is one static Go binary that turns your Ansible repositories into a fast, beautiful control center — web UI, full TUI and REST API included. No operator. No PostgreSQL. No message broker. Just your playbooks, visualized and runnable.

Why Pine

Everything AWX gives a small team.
None of what it demands.

AWX is a powerful platform — if you need enterprise RBAC across thousands of users and a workflow engine, use it. Pine is for everyone who just wants to see, understand and run their Ansible. Honestly compared:

Features

Point it at your repos. Pine does the rest.

Pine scans real Ansible repositories — no special project format, no surveys, no job templates to maintain.

Single static binary

One Go binary, zero runtime dependencies. Deploy with Docker, docker-compose, or scp it to a box and run it.

Multiple repos, auto-synced

Connect any number of Ansible repositories — git URLs or local paths. Pine keeps them fresh automatically.

Deep scanning

Playbooks, plays, tasks, roles — defaults, handlers, meta, dependencies — plus INI & YAML inventories, group_vars and host_vars.

Interactive topology graph

Your inventories rendered live — groups, children and hosts as an explorable graph, not a wall of INI.

Playbook task-flow

See every playbook as plays → roles → tasks, with tags, handlers and notify edges. Understand before you run.

Job engine with live streaming

Run playbooks with --check, --limit, --tags. Output streams live over SSE; history keeps a per-host ok / changed / failed recap. No Ansible installed? Simulation mode still works.

Web UI + TUI + REST API

An embedded web UI, a full terminal UI for SSH sessions, and a clean REST API for everything else. Same data, three doors.

No moving parts

State is plain JSON on disk. No external database, no Kubernetes operator, no RabbitMQ. Nothing to babysit at 3 a.m.

      Quickstart
      
Running before your coffee cools.

      
Pick your flavor. All three end at http://localhost:8743.

    

    

      

        
        
        
      

      

        

          

            
            
# pull & run — data persists in a named volume
$ docker run -d --name pine \
    -p 8743:8743 \
    -v pine-data:/data \
    -v ~/ansible:/repos/ansible:ro \
    ghcr.io/jgsqware/pine:latest

$ open http://localhost:8743

          

        

        

          

            
            
# docker-compose.yml
services:
  pine:
    image: ghcr.io/jgsqware/pine:latest
    ports:
      - "8743:8743"
    volumes:
      - pine-data:/data
      - ~/ansible:/repos/ansible:ro
volumes:
  pine-data:

$ docker compose up -d   # → http://localhost:8743

          

        

        

          

            
            
# grab the binary (or download a release)
$ go install github.com/jgsqware/pine@latest

# serve the web UI with the bundled Acme demo
$ pine serve --demo      # → http://localhost:8743

# or stay in the terminal
$ pine tui

          

        

      

    

  







  

    

      REST API
      
Everything the UI does, your scripts can too.

      
A small, predictable JSON API. List repos, inspect inventories, launch jobs, follow the SSE stream.
