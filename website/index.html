The Ansible control plane that doesn't need a control plane.

Pine is one static Go binary that turns your Ansible repositories into a fast, beautiful control center — web UI, full TUI and REST API included. No operator. No PostgreSQL. No message broker. Just your playbooks — visualized, planned before they run, and runnable.

Why Pine

Everything AWX gives a small team.
None of what it demands.

AWX is a powerful platform — if you need enterprise RBAC across thousands of users and a workflow engine, use it. Pine is for everyone who just wants to see, understand and run their Ansible. Honestly compared:

Features

Point it at your repos. Pine does the rest.

Pine scans real Ansible repositories — no special project format, no surveys, no job templates to maintain.

Plan mode

The terraform plan Ansible never had. Per task × host verdicts — run, skip or unknown with the exact missing variables — in milliseconds. See it below ↓

Drift detection

Pine periodically replays playbooks under --check and heat-maps where reality diverged from the repo, per playbook × host.

Plan-gated schedules

Cron-style schedules that refuse to fire blindly: if the plan fingerprint changed since you approved it, the run blocks until you review.

Light pipelines

Canary → approval gate → fleet, in three lines of YAML. Enough orchestration for rollouts without a workflow engine to feed.

Insights

Variable lineage, hygiene reports, run diffs and blast-radius analysis — static analysis over everything Pine already knows about your repo.

Facts, three ways

Built-in fact profiles (ubuntu-24.04, debian-12, rhel-9…), harvesting from real hosts via ansible -m setup, or pin values by hand.

Constructed inventories

Split inventory dirs merge in order and ansible.builtin.constructed is emulated — generated groups appear on the topology with badges.

Smart scanning

Recursive playbook and role discovery anywhere in the repo. Inventories detected by content — .yml, .yaml, .ini or no extension at all.

Topology & task-flow

Inventories as an explorable live graph — with a time-lapse over its git history — and every playbook as plays → roles → tasks → handlers.

Jobs, streamed live

Run with --check, --limit, --tags. Output streams over SSE; history keeps per-host recaps and feeds plan duration estimates.

Single static binary

One Go binary, state in plain JSON you can git-diff. No PostgreSQL, no operator, no broker. Nothing to babysit at 3 a.m.

Web UI + TUI + REST API

An embedded web UI, a full terminal UI for SSH sessions, and a clean REST API for everything else. Same data, three doors.

Plan mode

Plan before you apply.

The terraform plan Ansible never had. Pine statically evaluates every task against every host — no SSH, no side effects — and tells you what would run, what would skip and why, and exactly which variables it can't know yet. Then you fill in the blanks and re-plan, instantly.

pine plan acme-infra/ site.yml -i production estimated
Missing variables
ansible_facts.os_family ×28
ansible_facts.os_family =

Type a value, pick a fact profile or harvest real facts — Pine re-plans in milliseconds and the unknowns collapse into run / skip.

run 144 skip 12 unknown 28 exit 3 — unknowns present planned in 9 ms · ~4m 36s est. from past runs

Three-valued, honest

Every task × host gets a verdict: run, skip — with the condition that was false — or unknown, with the exact variable names Pine is missing. No guesses dressed up as certainty.

Facts, three ways

Built-in profiles (ubuntu-24.04, debian-12, rhel-9…), facts harvested from real hosts with ansible -m setup, or values you pin by hand in the missing-variables panel.

The whole shape of the run

Loop sizes, serial batches, which handlers would be notified, and an estimated duration learned from the timings of your past runs.

Exact mode

Want ground truth? Pine runs ansible-playbook --check through a JSON callback and renders it in the same UI — the badge flips from estimated to exact.

# plan from the CLI — same engine, same verdicts
$ pine plan acme-infra/ site.yml -e app_version=2.4.2 --profile ubuntu-24.04
run 762 · skip 138 · unknown 0   (9 ms)

# no facts pinned? unknowns make it exit 3 — a perfect CI gate
$ pine plan acme-infra/ site.yml && echo safe-to-merge
run 144 · skip 12 · unknown 28   → exit 3

      Operate
      
Day-2 operations, without the 3 a.m. surprises.

      
Service status shows which daemons are actually up. Drift detection tells you when reality wandered off. Plan-gated schedules refuse to fire blindly. Light pipelines roll changes out one careful step at a time.

    


    
    

      

        

          
Service status

          
Each host declares the services it runs (services: [teamcity-agent, docker]). Pine harvests their real state with ansible service_facts and draws a hosts × services heatmap — running, stopped, or unknown until checked. No agents to install.

      Quickstart
      
Running before your coffee cools.

      
Pick your flavor. All three end at http://localhost:8743.

    

    

      

        
        
        
      

      

        

          

            
            
# pull & run — data persists in a named volume
$ docker run -d --name pine \
    -p 8743:8743 \
    -v pine-data:/data \
    -v ~/ansible:/repos/ansible:ro \
    ghcr.io/jgsqware/pine:latest

$ open http://localhost:8743

          

        

        

          

            
            
# docker-compose.yml
services:
  pine:
    image: ghcr.io/jgsqware/pine:latest
    ports:
      - "8743:8743"
    volumes:
      - pine-data:/data
      - ~/ansible:/repos/ansible:ro
volumes:
  pine-data:

$ docker compose up -d   # → http://localhost:8743

          

        

        

          

            
            
# grab the binary (or download a release)
$ go install github.com/jgsqware/pine@latest

# serve the web UI with the bundled Acme demo
$ pine serve --demo      # → http://localhost:8743
$ pine service install   # …or run it as a systemd service

# plan before you apply — exit 3 if anything is unknown
$ pine plan ~/ansible site.yml --profile ubuntu-24.04
$ pine impact ~/ansible roles/nginx/templates/nginx.conf.j2

# or stay in the terminal — attach to the running daemon over its API
$ pine tui
$ pine attach           # drives `pine serve` / systemd / Docker

Everything the UI does, your scripts can too.
